# st-fetch — Fetching External Content into Cross

## What it does today

`st-fetch` retrieves a post from X (Twitter) by tweet ID using the Twitter v2
API and stores the raw response as a `data` entry in a `.json` container.

That's a narrow capability. The intent — fetching external content into the
Cross framework for analysis and improvement — is much broader and worth
designing deliberately.

---

## What it should do

The core value proposition of Cross is:

> Take any piece of writing, run it through the cross-product fact-check and
> analysis pipeline, improve it with AI, and publish a better version.

`st-fetch` is the **on-ramp** for content that was not generated by Cross.
It is the answer to: *"I already have a story — how do I get it into the system?"*

The gap today: `st-fetch` stores to `data[]` but a story needs to reach
`story[]` with `make`, `model`, `title`, `markdown`, `text`, `spoken`, and
`hashtags` fields before `st-fact`, `st-fix`, `st-merge`, and `st-post` can
work with it. The bridge is `st-prep`, which reads from `data[]` and writes
to `story[]`.

So the full pipeline for fetched content is:

```
st-fetch <source>  →  st-prep  →  st-fact / st-fix / st-merge / st-post
```

This is the same pipeline as generated content — which means fetched stories
are first-class citizens in every downstream tool once `st-prep` is run.

## Use cases

### 1. Refresh a published report

You wrote and published a report several years ago. The facts may be stale,
the framing dated, and new developments have occurred since.

```
st-fetch <tweet_id_or_url> old_report.json
st-prep old_report.json
st-bang old_report.json          # regenerate the topic with all 5 AI
st-cross old_report.json         # cross-product fact-check all versions
st-merge old_report.json         # synthesize best version
st-post --site myblog old_report.json
```

The original fetched story sits alongside the five freshly generated versions.
The cross-product fact-check will flag where the original is now outdated.
`st-fix` or `st-merge` produces the refreshed version.

### 2. Fact-check someone else's story

You read an article or post that makes specific claims you want to verify.
Fetch it, run the cross-product fact-check, and see how five independent AI
rate each claim.

```
st-fetch <tweet_id> article.json
st-prep article.json
st-fact --ai all -s 1 article.json
st-ls --story --fact article.json
```

### 3. Import your own archive

You have a collection of previously published stories in plain text or
markdown files. Bring them into Cross containers for fact-checking and
potential improvement.

```
st-fetch --file my_old_report.md archive.json --prep
st-cross archive.json
```

### 4. Competitive research

Fetch published reports on a topic from multiple sources (X posts, URLs).
Use Cross to compare their accuracy and identify consensus and disagreement
across sources.

### 5. Source validation before posting

Before publishing a story that cites external sources, fetch the source
material into the container alongside your draft. Run the cross-product
fact-check to verify citations are accurate.

## What st-fetch supports

| Source | Command | Needs API key? | Status |
|---|---|---|---|
| X (Twitter) tweet ID | `st-fetch <tweet_id> file.json` | Yes — `X_COM_BEARER_TOKEN` | ✅ |
| Plain text / markdown file | `st-fetch --file report.md file.json` | No | ✅ |
| Web page (URL) | `st-fetch --url https://... file.json` | No | ✅ |
| System clipboard | `st-fetch --clipboard file.json` | No | ✅ |
| Discourse post | `st-fetch --discourse <topic_id> file.json` | Yes — Discourse API key | roadmap |

`--prep` flag available on all sources — runs `st-prep` automatically after fetch.

## Data flow detail

`st-fetch` writes a `data` entry. The `make`/`model` pair encodes both the
origin type and the specific source — no separate metadata field is needed:

| Source | make | model |
|---|---|---|
| X (Twitter) | `url` | `twitter-v2` |
| Local file | `url` | `file` |
| Web page | `url` | `bs4` |
| Clipboard | `url` | `clipboard` |

`make = "url"` tells all downstream tools this story was not AI-generated.
Routing logic (`st-merge`, `st-fix` synthesize mode) falls back to the
user-specified AI when `make` is not in the AI list — no special casing needed.
The full source path/URL/tweet ID is preserved in `gen_response` for traceability.

Example data entry (clipboard source):

```json
{
  "make":         "url",
  "model":        "clipboard",
  "title":        "First line of pasted text",
  "text":         "Plain text content",
  "markdown":     "Plain text content",
  "tag":          "cross:url:clipboard",
  "gen_response": { "source": "clipboard", "raw": "..." }
}
```

`st-prep -d 1 file.json` reads that `data` entry and writes a `story` entry:

```json
{
  "make":     "url",
  "model":    "bs4",
  "title":    "Cleaned title",
  "markdown": "Formatted markdown + reading tagline",
  "text":     "Plain text",
  "spoken":   "TTS-ready text",
  "hashtags": ["#topic", ...],
  "fact":     []
}
```

After `st-prep`, the fetched story is indistinguishable from a generated story
in every downstream tool.

## Required configuration

`--file`, `--url`, and `--clipboard` sources require no API keys.

For the tweet ID source, add to `.env`:

```
X_COM_BEARER_TOKEN=<your bearer token>
```

Obtain from [developer.x.com](https://developer.x.com). A free account with
Basic access (read-only) is sufficient for tweet fetching.
